## 1. 当前项目阶段（固定基线）

当前阶段按以下基线理解，不自行改写：

1. 低频体系第一阶段已基本完成
   - Job 1–10A 已完成或完成主体
   - lowfreq framework 已建立
   - Universe（A / B / C）已建立
   - 低频采集已从“全市场模式”切到“Universe 驱动模式”

2. 接下来研发顺序固定为
   - 先做中频
   - 再做高频

3. 中频和高频必须延续低频逻辑
   - 不允许另起炉灶
   - 不允许并行造第二套 ingestion/runtime 体系

## 2. 总体研发原则

### 2.1 Universe 规则

- A：高频
- B：中频
- C：低频
- 强制关系：`A ⊂ B ⊂ C`

### 2.2 采集边界

- 低频只读 C
- 中频只读 B
- 高频只读 A
- 不允许回到全市场采集
- 不允许绕过 `symbol_universe`

### 2.3 架构继承关系

中频 / 高频必须尽量复用低频已有能力：

- `symbol_universe`
- raw / source mirror
- canonical current
- history / version
- daemon / state
- summary / docs / runbook

原则上：
- 低频是地基
- 中频是在低频框架上加更快节奏的数据层
- 高频是在中频/低频一致性框架上加更强资源约束与更细颗粒度执行

### 2.4 中频与高频的基本区别

- 低频：慢变量、文档、基础表、资产层
- 中频：B list 上的日内更新 / 结构化行情 / 板块与重点个股状态更新
- 高频：A list 上的更高频、更精细、更强资源约束的数据层

### 2.5 非分叉原则

中频 / 高频不得：
- 新造一套独立 Universe 体系
- 新造一套独立 current/history/version 语义
- 新造一套与 lowfreq 不兼容的 run state / daemon state
- 直接回退到全市场抓取

## 4. Job 详细拆分

## B1：中高频总体研发计划与上下文固化

### 目标
- 写清 iFA 从低频进入中频 / 高频的整体研发路径
- 固化统一上下文、边界、命名与顺序
- 解决后续 session 容易跑偏的问题

### 边界
- 只做规划、文档、上下文固化
- 不开发任何新的 dataset
- 不修改 daemon / ingest 逻辑

### 输入
- 已完成的 lowfreq framework
- 已建立的 A / B / C Universe 规则

### 输出
- `IFA_MID_HIGH_PLAN.md`
- 更新后的 `IFA_PROJECT_CONTEXT.md`
- developer workspace 内的同步归档文件

### 验收标准
- B1–B5 / C1–C4 拆分明确
- 明确中频先于高频
- 明确 A/B/C 的使用边界
- 明确中频 / 高频必须继承低频逻辑

### 非目标
- 不开发中频采集
- 不开发高频采集
- 不做 facts / signals

## B2：中频需求与数据范围定义

### 目标

- 明确 B list 的观测目标和数据范围

### 边界
- 只定义需求与范围
- 不落代码
- 不接入新数据源

### 输入
- A / B / C Universe 规则

### 验收标准
- 明确中频覆盖对象仅限 B list
- 明确每类数据的用途、频率、来源候选、数据形态

- 不做调度实现
- 不做高频设计

## B3：中频技术方案与调度 / 版本策略

### 目标
- 设计中频在现有 lowfreq 框架上的继承方式
- 定义调度、状态、版本、保留、资源控制策略

### 边界
- 方案优先，尽量复用 lowfreq 现有组件
- 不大改底层架构
- 不落地完整中频 dataset

### 输入
- B2 的需求与范围定义
- 低频已有的 symbol_universe / raw / current / history / version / daemon / state 能力

### 输出
- 中频技术方案文档
- 调度与版本策略说明
- 中频命名 / 表层 / 运行层约定

### 验收标准
- 明确中频如何复用现有框架
- 明确 current/history/version 是否复用原语义
- 明确 daemon / state 是复用、扩展还是分层
- 明确资源预算与运行保障最小要求

### 非目标
- 不写完整生产实现
- 不引入高频资源策略
- 不扩中频 dataset 本体
