# IFA_ARCHIVE_LAYER_EXECUTION_PLAN.md

> 文档性质：归档层执行路线文档。
>
> 文档目标：把 Archive Layer 的正式设计，整理为后续研发和执行的固定路线图。
>
> 使用方式：后续归档层研发必须按本文件顺序推进，不跳阶段、不提前混入后置项。

---

## 1. 总体执行原则

Archive Layer 的后续推进必须遵守以下原则：

1. 先把文档固化，再进入实现。
2. 先做框架，再做资产层扩展。
3. 先让 Archiver 跑起来，再补历史数据。
4. 先做原始数据归档，不提前做生产 snapshot/version 沉淀。
5. Runtime Daemon 和 Archiver 职责分离，不能混用。
6. 不重新设计总体方向，不偏离当前正式基线。

固定执行顺序：
- D1
- D2
- D3
- D4
- D5
- D6
- D7

---

## 3. D1–D7 分阶段执行路线

### D1：归档框架

#### D1 做什么
- 建立 Archiver 基础框架
- 建立归档运行控制表：
  - `archive_jobs`
  - `archive_runs`
  - `archive_checkpoints`
- 建立 Archive Layer 的 summary / health / watchdog 机制
- 建立夜间两段窗口：
  - 21:30–22:30
  - 02:00–03:00

#### D1 的目标
- 先让 Archiver 作为系统跑起来
- 先有任务定义、运行日志、checkpoint、health
- 先有夜间执行框架与状态持久化

#### D1 不做什么
- 不做大面积历史补齐
- 不做生产 snapshot/version 沉淀
- 不做高频相关实现
- 不做 fancy 报表层

#### D1 完成标志
- Archiver 可以在夜间窗口运行
- 每次运行都有落库记录
- checkpoint 可以被记录和查询
- health / watchdog 可以查询
- 运行状态与 Runtime Daemon 状态不混淆

---

### D2：股票历史归档

#### D2 做什么
- 股票日线归档
- 股票相关资产类历史归档
- 各板块前 10–30 活跃股的历史补齐
- 如有需要，补股票分钟历史归档

#### D2 的目标
- 先把股票资产层做出来
- 让 Archive Layer 真正开始沉淀长期股票资产

#### D2 的范围控制
- 不做全市场无限抓
- 采用当前固定规则：
  - 各板块前 10–30 活跃股
  - 默认先按前 10
  - 需要时再扩到前 20 / 前 30

#### D2 不做什么
- 不扩到宏观
- 不扩到商品/期货/贵金属
- 不提前做生产快照沉淀

#### D2 完成标志
- 股票类归档任务可夜间持续推进
- 有 checkpoint / resume
- 有批次推进记录
- 有基础历史资产表可查询

---

### D3：宏观历史归档

#### D3 做什么
- 利率
- CPI / PPI
- PMI
- GDP
- 工业增加值
- 社融 / M2
- 其他重要慢变量

#### D3 的目标
- 建立宏观慢变量资产层
- 让未来宏观回看、比较分析有长期资产基础

#### D3 的原则
- 宏观不需要过度裁剪
- 只要有价值、能拿到、值得沉淀，就应纳入

#### D3 完成标志
- 宏观数据有自己的 archive_jobs / runs / checkpoints
- 可以按资产类别查看回填进度
- 宏观历史资产可长期查询

---

### D4：商品 / 期货 / 贵金属历史归档

#### D4 做什么
- 合约主表
- 日线
- 分钟历史（如需要）
- 仓单
- 持仓排名
- 结算参数
- 重点合约历史

#### D4 的目标
- 建立商品 / 期货 / 贵金属长期资产层

#### D4 的范围控制
- 保持重点池
- 不超过 30 个重点对象
- 以热门品种 / 主力合约为主

#### D4 完成标志
- 商品/期货/贵金属归档链路能夜间持续跑
- 有 checkpoint / resume
- 有长期历史数据表与运行日志

---

### D5：独立运行

#### D5 做什么
- Archiver 脱离 OpenClaw 会话依赖
- Archiver 可以自己持续运行
- main 可以接管监控

#### D5 的目标
- Archiver 从“开发中工具”进入“正式运行系统”

#### D5 关键要求
- 不依赖当前聊天上下文才能执行
- 有固定启动方式
- 有固定状态查看方式
- 可被 main 或外部监控系统接管

#### D5 完成标志
- Archiver 可以独立运行
- 主会话可查询其健康与最近运行结果
- 不需要人工盯着单次脚本跑完

---

### D6：全量测试与验收

#### D6 做什么
- 整体验收
- checkpoint / resume 测试
- 夜间窗口测试
- 数据一致性测试
- summary / health / watchdog 测试

#### D6 的目标
- 从“代码存在”变成“真正可运行系统”

#### D6 关注点
- checkpoint 是否可靠推进
- 失败后是否能从断点恢复
- 夜间两段窗口是否按预算执行
- run summary 是否完整落库
- health / watchdog 是否能真实反映状态
- 数据补齐是否一致、是否去重、是否可重复运行

#### D6 完成标志
- Archiver 被证明可持续运行
- 关键场景已完成测试
- blockers 与限制项被清晰列出并可跟踪

---

### D7：生产快照资产沉淀（后置）

#### D7 做什么
- 低频 version 的长期归档
- 中频 snapshot 的长期归档
- 未来 report-time snapshot 的长期归档

#### D7 的目标
- 让生产层关键时点结果具有长期回放与对比能力

#### 为什么 D7 必须后置
因为这件事依赖前提：
- lowfreq 本身足够稳定
- midfreq 本身足够稳定
- Runtime 的 version / snapshot 语义已经足够清晰

如果前置做 D7，风险是：
- 把生产层当前不稳定语义固化进 Archive Layer
- 让资产层被运行层补丁/临时逻辑污染

#### D7 完成标志
- 生产快照沉淀有独立表设计
- 有明确来源、时点和长期回溯路径
- 与原始归档资产层语义清晰分离
