## 1. A / B 的总体定位

### 1.1 A（低频）是什么
A（这里指低频层，不是 Universe A）是当前 collection 系统中的慢变量、基础表、资产层与低速更新层。

它负责的核心内容包括：
- 基础主数据
- 公告 / 文档 / 资产类数据
- 慢变量
- 中长期可复用的基础资产表
- 为中频和未来高层分析提供稳定底座

低频的本质是：
- 面向稳定资产
- 面向较低更新频率
- 面向基础沉淀
- 面向“先把地基打牢”

它不追求交易日内频繁刷新，而是强调：
- 正确性
- 完整性
- 可复用性
- 作为中频和后续分析层的底座价值

### 1.2 B（中频）是什么
B（这里指中频层，不是 Universe B）是当前 collection 系统中的窗口化更新层。

它负责的核心内容包括：
- 面向重点对象的日内窗口采样
- 结构化市场状态更新
- 当天报告前的数据稳定化
- 重点股票 / 板块 / 资金 / 结构数据的窗口级刷新

中频的本质是：
- 面向窗口
- 面向结构化市场状态
- 面向当天报告支持
- 面向“重点对象的时效性更新”

中频不是实时系统，不是高频流系统，也不是无限轮询系统。它强调的是：
- 固定窗口
- 预热 + 收口
- snapshot / current 的时点稳定性
- 报告使用前的冻结/确认

### 1.3 为什么要分这两层
低频与中频必须分层，原因不是为了拆成两套分裂系统，而是为了把两类本质不同的任务明确区分：

低频解决：
- 什么数据应该长期稳定存在
- 什么主数据、文档、基础表要先打底
- 什么慢变量要持续沉淀

中频解决：
- 哪些重点对象需要交易日内多窗口刷新
- 哪些数据要为盘前 / 午间 / 收盘后报告服务
- 哪些市场状态需要在窗口内形成稳定 snapshot

如果不分层，会导致：
- 慢变量任务和窗口任务互相污染
- 资源预算混乱
- 版本语义混乱
- 低频和中频都失去清晰边界

### 1.4 它们分别服务什么目的
低频（A）服务的主要目的：
- 打底座
- 做基础资产沉淀
- 提供长期稳定表层
- 为中频与后续分析提供基础材料

中频（B）服务的主要目的：
- 为当天报告与窗口判断提供更新后的重点数据
- 在交易日内形成结构化市场状态
- 把重点对象的数据更新到报告可用状态

因此，两者不是竞争关系，而是：
- 低频提供地基
- 中频提供窗口态更新

## 2. Runtime Daemon 统一视角

### 2.1 为什么 A 和 B 应视为同一个 Runtime Daemon 体系下的不同 group
当前明确方向是：

**低频（A）和中频（B）不再视为两套分裂系统，而应统一理解为同一个 Runtime Daemon 体系下的不同 groups。**

这样理解的原因包括：
1. 两者本质上都属于生产采集层。
2. 两者都服务于当前运行时，而不是长期资产沉淀层。
3. 两者都需要统一的 run state / summary / watchdog / health 设计范式。
4. 两者都应复用相同的数据库、schema、基本运行语义。
5. 两者的差异主要体现在：
   - 频率不同
   - 窗口不同
   - 数据目标不同
   - version/snapshot 侧重不同

因此，从产品与系统视角看，最合理的抽象不是“两个独立系统”，而是：
- 一个统一的 Runtime Daemon 体系
- 内部包含 lowfreq groups
- 内部包含 midfreq groups
- 将来如果需要，也可在同一体系下规划 future highfreq groups（当前不实现）

### 2.2 lowfreq / midfreq 当前已有实现如何被统一理解
当前已有实现可以统一理解为：

#### lowfreq 当前实现
- 已具备 daemon
- 已具备 group
- 已具备 current/history/version
- 已具备 state / health / runs
- 主要承担低速与底层资产采集

#### midfreq 当前实现
- 已具备独立 daemon 与窗口配置
- 已具备 current/history/version
- 已具备 summary / watchdog / health
- 主要承担窗口采样与重点对象刷新

统一理解以后，可以把它们看成：
- lowfreq：Runtime Daemon 中的低频 groups
- midfreq：Runtime Daemon 中的中频 groups

也就是说，当前已有实现并不是被推翻，而是被放回一个更统一的产品视角里理解。

### 2.3 尽量少改现有实现，而不是推倒重来
当前设计原则不是推倒重来，而是：
- 尽量少改现有实现
- 优先统一认知与上层设计
- 逐步收敛为统一 Runtime Daemon 体系

这样做的原因是：
1. lowfreq 与 midfreq 已经各自有可运行基础。
2. 当前更大的问题不是“完全没有系统”，而是“认知和规划容易跑偏”。
3. 先把统一设计写清楚，再逐步对齐实现，比重构一切更稳。

因此，本阶段的产品设计目标不是“代码重写”，而是：
- 统一产品视角
- 统一运行视角
- 统一后续任务拆解

## 3. Universe 与业务关注层

### 3.1 A / B / C Universe 是采集边界
当前必须明确：

A / B / C Universe 的角色是：
**采集边界。**

它们的意义在于：
- 限定不同频率层允许读取的对象范围
- 防止回到全市场无限抓
- 形成稳定、可控的资源边界

当前正式规则：
- A Universe：高频采集边界（未来）
- B Universe：中频采集边界
- C Universe：低频采集边界
- 强制关系：`A ⊂ B ⊂ C`

Universe 解决的问题是：
- “这一层最多可以采谁”

它不等于：
- 业务今天最关心谁
- 哪些对象应该进入常驻重点关注
- 哪些对象是短期动态热点

### 3.2 重点关注（Key Focus）与关注列表（Watchlist）是业务层
重点关注（Key Focus）与关注列表（Watchlist）属于：
**业务层。**

它们解决的问题不是采集边界，而是：
- 今天 / 当前阶段最值得关注的是谁
- 哪些对象需要更密切观察
- 哪些板块 / 标的需要被长期在线关注
- 哪些对象是动态发现出来的热门池

因此：
- Universe = 采集边界层
- Focus / Watchlist = 业务关注层

这两层不能混成一个概念。

### 3.3 为什么“重点关注 / 关注”必须数据库化
重点关注 / 关注列表必须数据库化，原因包括：
1. 业务关注是会变化的，不适合硬编码在代码里。
2. 不同时间、不同板块、不同策略下，重点关注对象会变化。
3. 后续需要审计：某一天为什么关注这些对象，来源是什么，何时加入、何时移除。
4. 后续需要多来源融合：人工指定、规则筛选、动态发现、长期常驻。
5. 低频与中频都需要消费同一层业务关注对象，而不是各自维护一份分裂列表。

### 3.4 为什么不能只靠硬编码列表
不能只靠硬编码列表，原因在于：
- 硬编码不可审计
- 硬编码难维护
- 硬编码难复用
- 硬编码无法支持动态发现池
- 硬编码无法支持多类 focus lists 共存
- 硬编码会让 lowfreq / midfreq / archive / future report 读到不同版本的“关注对象定义”

因此，业务关注层必须数据库化。

## 4. 重点关注 / 关注列表的数据库化

### 4.1 为什么需要业务层 focus tables
需要业务层 focus tables 的原因是：
- Universe 只是边界，不是业务关注逻辑
- 业务关注需要长期池 + 动态池
- 低频与中频都需要消费相同的业务关注定义
- 科技板块专项需要“常驻关注池 + 动态发现池”并行存在

因此，必须把 focus layer 作为正式设计的一部分。

### 4.2 建议的数据表
建议至少引入两张业务层表：
- `focus_lists`
- `focus_list_items`

### 4.3 focus_lists
`focus_lists` 用来定义“这是一类什么关注池”。

建议包含的信息例如：
- list_id
- list_name
- list_type（常驻 / 动态 / 板块专项 / 宏观专项 / 商品专项等）
- asset_type（股票 / 板块 / 宏观 / 期货 / 商品 / 贵金属）
- scope_desc
- source_type（manual / rule / discovered）
- enabled
- created_at
- updated_at

其作用是定义：
- 这份关注池的身份
- 这份关注池的目的
- 这份关注池由什么机制产生

### 4.4 focus_list_items
`focus_list_items` 用来定义“这份关注池里具体有哪些对象”。

建议包含的信息例如：
- item_id
- list_id
- object_type（symbol / sector / macro_series / futures_contract 等）
- object_id / symbol / sector_code
- rank
- weight
- reason
- source_ref
- is_active
- created_at
- updated_at

其作用是定义：
- 某个关注池中具体有哪些对象
- 排名是多少
- 为什么进池
- 是否仍有效

### 4.5 它们和 symbol_universe 的关系
`focus_lists` / `focus_list_items` 与 `symbol_universe` 的关系应理解为：

- `symbol_universe`：定义“允许采集边界”
- `focus_lists`：定义“当前业务关注对象”

它们不能互相替代。

正确关系是：
- Universe 决定能不能采
- Focus 决定应该优先关注谁

例如：
- 某股票不在 C Universe，就不应进入 lowfreq 正式采集
- 某股票虽然在 C Universe，但未必进入当前重点关注池
- 某股票进入动态热点池，也仍然要受 Universe 采集边界约束

### 4.6 它们如何服务低频和中频
对 lowfreq：
- focus tables 用于帮助决定哪些主题、板块、代表股要优先沉淀静态或慢变量资产
- 尤其对科技板块、商品重点池等有帮助

对 midfreq：
- focus tables 用于帮助决定哪些对象在窗口内更值得采样或扩展跟踪
- 尤其适用于科技热门板块、动态热点、代表股窗口更新

因此，focus tables 是 A/B 一体化设计中的正式一层，而不是“以后再说的附加功能”。

## 5. 科技板块专项

科技板块专项必须作为正式设计写入 A/B 体系。

### 5.1 固定科技主线
固定科技主线指：
- 系统长期在线关注的科技主线
- 长期常驻，不随短期热点完全漂移
- 每条主线默认不超过 5 个代表股

这类对象的特点是：
- 稳定
- 长期有价值
- 适合做常驻关注池

### 5.2 热门科技板块发现层
热门科技板块发现层指：
- 根据市场热度、活跃度、板块变化持续发现热门科技板块
- 不是长期不变，而是动态变化

这一层的意义是：
- 补充常驻科技主线的不足
- 反映市场阶段性热点
- 为中频窗口采样和低频补充沉淀提供动态输入

### 5.3 每个热门板块前 10 / 前 20 活跃股
对每个热门科技板块，应进一步抽取该板块内的前 10 / 前 20 活跃股。

这样做的原因是：
- 不需要全板块无限抓
- 但也不能只看 1–2 个代表股
- 需要在“板块级观察”和“个股级观察”之间形成合理桥梁

默认建议：
- 先按前 10
- 有需要再扩到前 20

### 5.4 常驻关注池 vs 动态发现池
科技板块专项必须同时存在两层：

#### 常驻关注池
- 长期固定在线
- 稳定追踪
- 默认代表长期主线

#### 动态发现池
- 随市场变化更新
- 反映当前热点
- 用于补足常驻池的静态性

### 5.5 为什么这两层都必须存在
如果只有常驻关注池：
- 系统会过于静态
- 无法反映新热点

如果只有动态发现池：
- 系统会失去长期主线连续性
- 不利于长期跟踪与对比分析

因此，科技板块专项必须设计成：
- 常驻池 + 动态池 并行

## 6. 低频（A）具体采集范围

### 6.1 股票低频
股票低频主要包括：
- 基础主数据
- 公告 / 文档 / 资产类表

### 6.2 宏观低频

- 利率

- PMI
- GDP
- 工业增加值
- 社融 / M2

这些天然属于低频层，因为：
- 更新频率低
- 更适合作为底层慢变量资产
- 不需要交易日内窗口化刷新

### 6.3 商品 / 期货 / 贵金属低频

- 慢变量参数

### 6.4 科技板块低频

- 固定科技主线清单
- 板块静态成分关系
- 长期代表股基础信息
- 用于支持科技主线长期观察的慢变量资产

## 7. 中频（B）具体采集范围

### 7.1 股票中频
股票中频主要包括：
- 个股日线窗口更新

- 北向 / 主力资金 / 龙虎榜 / 涨跌停结构等

宏观中频主要包括：
- 需要在盘前 / 午间 / 收盘后更新的窗口型宏观代理数据
- 更偏市场窗口观察而非长期慢变量的宏观状态数据

### 7.3 商品 / 期货 / 贵金属中频
商品 / 期货 / 贵金属中频主要包括：
- 重点品种 / 主力合约的窗口型状态更新
- 日级或窗口级的价格 / 结构 / 活跃度更新
- 但不进入高频流式层

### 7.4 科技板块中频
科技板块中频主要包括：
- 热门科技板块动态热点跟踪
- 常驻科技主线代表股窗口更新
- 每个热门板块前 10 / 前 20 活跃股的窗口跟踪

## 8. 时间与窗口

### 8.1 北京时间作为业务标准
A/B Runtime 设计中的业务时间标准统一以：
**北京时间（Asia/Shanghai）**
为业务标准。

### 8.2 低频窗口
当前低频窗口重点包括：
- `daily_light`
- `weekly_deep`

其特征是：
- 频率低
- 更偏日常 / 周度
- 更适合慢变量与资产类任务

### 8.3 中频窗口
当前中频窗口重点包括：
- prewarm_early
- pre_open_final
- midday_prewarm
- midday_final
- post_close_prewarm
- post_close_final
- night_settlement

### 8.4 预热窗口 / 收口窗口
中频应明确区分：
- 预热窗口
- 收口窗口

预热窗口负责：
- 准备
- 健康检查
- 状态刷新
- 轻量预取

收口窗口负责：
- 最终补漏
- 状态确认
- snapshot 稳定化
- 报告前读取准备

## 9. Version / Snapshot 原则

### 9.1 什么数据要 version
更适合 version 的对象包括：
- 低频正式资产层的版本化结果
- 需要明确 active / candidate / superseded 语义的稳定资产对象
- 报告前明确冻结口径的对象

### 9.2 什么数据不要 version
不适合默认版本化的对象包括：
- 纯原始历史流水数据
- 以 merge / upsert / 去重为主的历史事实层
- 不需要“active version”语义的普通历史记录

### 9.3 中频为什么更偏 snapshot
中频更偏 snapshot，是因为：
- 它是窗口化系统
- 它服务的是某一时点的市场状态
- 它的核心价值在于“这个窗口结束时的数据状态是什么”

因此，中频比低频更适合强调：
- snapshot
- freeze point
- 当前窗口的稳定口径

### 9.4 无变化不建新 version
无论低频还是中频，必须坚持：
- 无变化 → 不建新 version
- 只记录 run 成功与 freshness

这样可以避免：
- version 噪声膨胀
- 历史语义变脏
- 运行成功与内容变化混淆

## 10. 运行报告 / summary / watchdog

### 10.1 A 和 B 如何落库
A/B 都必须有正式落库运行记录。

低频（A）至少应有：
- runs
- group state
- summary 语义
- health / watchdog

中频（B）至少应有：
- execution summary
- daemon state
- watchdog / health
- 窗口执行结果

### 10.2 怎么查
必须支持：
- 查 latest run
- 查 group state
- 查 latest summary
- 查 daemon health
- 查 freshness

### 10.3 报告如何生成
A/B 都应生成业务可读的运行报告 / summary，而不是只面向开发者日志。

运行报告至少应回答：
- 昨天 / 今天是否跑了
- 跑了哪些 group
- 哪些成功
- 哪些失败
- 当前 blocker 是什么
- 是否健康

### 10.4 为什么要做成业务可读
因为：
- collection 层最终是为业务服务，而不是只为工程自娱自乐
- 业务方关心的不是 stdout，而是“现在是否健康、今天能否用、哪里有问题”
- 只有业务可读 summary，系统才能被真正运营起来

## 11. 一句话总目标

低频（A）+ 中频（B）应被统一理解为同一个 Runtime Daemon 体系下的不同 groups：低频负责打底和慢变量资产层，中频负责窗口化重点更新；Universe 管采集边界，focus tables 管业务关注层，科技板块采用常驻池 + 动态池双层设计，后续研发在这一统一设计基线上继续推进。
