6. What each worker writes
lowfreq:
- slow/reference/fundamental/news/calendar data
- examples: trade calendar, stock/index/fund basics, announcements, news, research reports, investor QA, holdings/pledge/reference truth

midfreq:
- daily/post-session market-structure data
- examples: equity/index/ETF daily bars, flows, turnover, limit-up detail, dragon tiger list, sector performance

highfreq:
- intraday working data
- examples: stock/index/proxy 1m working, futures minute working, auction snapshots, event stream
- derived working: breadth, heat, leader candidates, signal state, limit-event state

archive:
- archive runs/checkpoints/catch-up state
- retained history layers
- structured archive output snapshots

7. What the key tables mean
current tables:
- latest canonical state for a dataset
history tables:
- retained time-series / historical preservation path
working tables:
- highfreq operational/intraday working state, not long-term truth-of-record
checkpoint/catch-up tables:
- archive progression/control truth
runtime tables:
- schedule, worker state, run evidence, governance/watchdog truth

8. Whether watchdog exists and what it actually does
What exists:
- timeout-aware overlap governance
- timeout marking
- stale active-run / stale heartbeat visibility
- operator-readable watchdog state in unified daemon status

What does not yet exist:
- separate external process supervisor that force-kills a hung inline worker process
- separate restart daemon/reaper for zombie subprocesses

So the truthful statement is:
- watchdog/governance EXISTS at the DB/operator layer
- hard external process recovery does NOT yet exist

9. Remaining truthful limitations
- no external hard-kill supervisor yet
- 1m/15m archive remains practical forward-only behavior
- sector_performance is now usable, but as a post-close daily midfreq dataset
- status command logs midfreq dataset registration noise on startup
