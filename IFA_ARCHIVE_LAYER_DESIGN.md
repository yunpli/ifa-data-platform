# IFA_ARCHIVE_LAYER_DESIGN.md

> 文档性质：归档层（Archive Layer）完整产品设计正式基线文档。
>
> 文档目的：把归档层作为一个完整产品/系统定义清楚，固化其定位、运行逻辑、数据库方向、监控要求、阶段划分与后续执行顺序。
>
> 适用范围：本文件用于指导 iFA 归档层后续 D1–D7 的设计实现、运行治理、验收与长期维护。
>
> 约束说明：本文件以当前已拍板口径为准，不在本文件内重新讨论总体方向，不在本文件内引入高频实现，不把生产快照资产沉淀提前到第一阶段。

---

## 1. 归档层的定位

归档层不是低频/中频的附属工具，也不是“把今天的数据再存一份”。

归档层的定位是：

**把更大范围、更长时间、更完整的数据，沉淀成 iFA 自己的长期资产库。**

归档层要解决的问题包括：

1. 低频和中频只覆盖重点对象，覆盖范围不够广。
2. 运行层数据不会长期保留，无法作为长期资产层直接使用。
3. 未来要做画图、技术分析、回溯、比较分析，没有长期资产库无法稳定支撑。
4. 生产层形成的 snapshot / version 将来需要有长期归宿，不能永远停留在运行态表里。

因此，归档层的核心属性是：
- 面向长期资产
- 面向历史补齐
- 面向未来分析
- 不以当天报告时效为核心

换言之，归档层的目标不是“今天能不能出报告”，而是“明天、下个月、明年是否还有一套可回溯、可对比、可分析的长期资产库”。

---

## 2. 归档层与运行层的关系

当前系统应明确分成两类运行系统：

### A. Runtime Daemon
Runtime Daemon 负责生产采集层。

内部可包含：
- lowfreq group
- midfreq group
- future highfreq group

其职责是：
- 在合适窗口抓取重点数据
- 为当天报告服务
- 形成 current / snapshot / summary
- 维护运行态 freshness、watchdog、窗口执行与当日生产闭环

Runtime Daemon 的本质是：
- 面向“今天”
- 面向“窗口”
- 面向“报告时效”
- 面向“重点对象”

### B. Archiver
Archiver 负责资产沉淀层。

其职责是：
- 补历史
- 扩范围
- 做 checkpoint / resume
- 写长期归档
- 不和生产窗口抢资源

Archiver 的本质是：
- 面向“长期”
- 面向“补齐与沉淀”
- 面向“资产库”
- 面向“未来分析使用”

### 两者关系
两者关系必须明确：
- **Runtime Daemon：为今天服务**
- **Archiver：为长期服务**

它们不是一套系统里的两个可互相替代的模式，而是两个目标不同、资源策略不同、状态语义不同的运行层：
- Runtime Daemon 解决当前生产问题。
- Archiver 解决长期资产问题。

因此，后续实现中必须避免：
- 用 Runtime Daemon 去承担长期大回填任务
- 用 Archiver 去承担当天报告前的数据生产责任

## 3. 归档层的数据原则

### 3.1 原始数据优先重新采
归档层默认应当自己重新采集原始数据，而不是优先从生产层倒数据。

原因如下：
1. 生产层数据范围较窄，只覆盖重点对象。
2. 生产层数据带有窗口裁剪语义，不等于长期资产的完整原始历史。
3. 生产层可能带测试/补丁/临时数据，直接倒库容易把运行层污染带进资产层。
4. 归档层本身的目标就是建立长期、可校正、可补齐的数据资产库，因此需要独立建立自己的原始采集链路。

因此，下面这些类型默认由归档层自己采：
- 日线
- 分钟线
- 宏观值
- 期货值
- 板块值
- 其他未来用于长期分析的原始市场数据

总结为一句话：

**原始市场数据，归档层优先自行重采。**

### 3.2 生产快照 / 版本产物要沉淀（后置）
有些对象不能靠“重新采原始数据”恢复完全相同的语义，例如：
- 某个中频窗口冻结后的市场状态
- 某个低频运行时的 version 状态
- 某次报告生成前使用的 snapshot

这些对象不是单纯的“原始市场值”，而是生产层在某个时间点形成的特定运行结果、冻结结果或使用结果。

因此，这类对象后续应当从生产层沉淀到资产层。

但这里必须明确优先级：
- 这件事是**后置项**
- 不是归档层第一阶段的核心目标
- 不应在 D1–D6 阶段过早混入实现

总规则固定为：
- **原始市场数据：归档层自己采**
- **生产快照 / version 产物：等 lowfreq 和 midfreq 稳定后，再沉淀过去**

## 4. 归档层的覆盖范围

归档层覆盖四大类资产：

### 4.1 股票
股票归档层至少包括：
- 日线
- 分钟历史
- 低频资产类表
- 中频结构类历史
- 科技板块相关历史
- 各板块活跃股历史

这里的股票归档层不是只为了“今天看几只重点股”，而是为了形成一个可持续积累、可长期回看、可用于绘图和比较分析的股票资产历史层。

### 4.2 宏观
宏观归档层至少包括：
- 利率
- CPI / PPI
- PMI
- GDP / 工业增加值
- 社融 / M2
- 其他重要慢变量

宏观层的目标不是当天交易时效，而是形成能够长期支撑宏观背景判断、横向比较、周期分析的慢变量资产层。

### 4.3 商品 / 期货 / 贵金属
商品与期货归档层至少包括：
- 合约主表
- 日线
- 历史分钟
- 仓单
- 持仓排名
- 结算参数
- 重点合约历史

贵金属作为商品/期货体系中的重点方向，也纳入这一类资产归档。

### 4.4 科技板块专项
科技板块归档层至少包括：
- 科技板块清单
- 科技板块成分股历史
- 热门科技板块代表股历史
- 长期在线科技主线相关历史数据

科技板块专项的意义在于：
- 它不是一次性热点表
- 而是长期主线资产的历史沉淀层
- 未来可支持画图、对比、主线演化分析与长期回看

## 5. 归档层的范围控制

归档层不是“全市场无限抓”，而是**有控制地扩大**。

### 5.1 股票范围控制
股票归档层不再采用“热门前 1000 个股票”这种粗放口径。

当前正式规则为：
- 按各板块前 10–30 个活跃股来做
- 默认先按前 10
- 有需要再扩到前 20 / 前 30

也就是说：
- 不是从当前 Runtime 的重点池直接拷贝
- 也不是全市场铺开
- 而是按板块视角做有控制的扩面

后续仍需单独定义“活跃”的量化标准，但本文件先固定范围控制原则。

### 5.2 商品 / 期货 / 贵金属范围控制
商品 / 期货 / 贵金属继续保持重点池思路：
- 不超过 30 个重点对象
- 以热门品种 / 主力合约为主

这样可以确保：
- 工程量可控
- 夜间窗口可控
- checkpoint / resume 可控
- 资产层先把高价值对象沉淀起来

### 5.3 宏观范围控制
宏观层不需要过度裁剪。

原则是：
- 只要有价值
- 能拿到
- 值得沉淀

就应纳入宏观归档层。

原因很简单：
- 宏观数据量不像分钟线那样爆炸
- 宏观数据天然就是慢变量资产
- 裁得太狠反而会损失未来分析能力

---

## 6. 归档层的时间范围

归档层第一阶段固定为：

**回填过去一年。**

这样设定的原因是：
1. 足够支持画图、技术分析、比较分析。
2. 足够支持未来多数 2.x / 3.x 回看场景。
3. 工程量可控，不会一开始就把系统拖入超长历史工程泥潭。
4. 后续若要扩更长历史，只需要扩配置与补跑，不需要推翻当前设计。

因此，第一阶段不是要一次性做“所有历史”，而是建立一个：
- 可跑通
- 可断点续跑
- 可扩年限
- 可长期维护

的归档框架和资产库基线。

---

## 7. 归档层的运行时间

归档层采用夜间两段运行：
- 窗口 1：21:30–22:30
- 窗口 2：02:00–03:00

固定规则：
- 每次最多 1 小时
- 每天两次
- 总预算约 2 小时
- 不得与 lowfreq / midfreq 生产窗口冲突

这样设计的好处包括：
1. 第一轮失败，第二轮还有机会补跑。
2. 可以把不同资产类别拆开跑。
3. 更利于 checkpoint / resume。
4. 不会长时间占用资源，不会和 Runtime Daemon 争抢当日生产能力。

因此，归档层的夜间运行不是“尽量多跑”，而是：
- 有预算
- 有分段
- 有节奏
- 有补救机会

## 8. 归档层的版本与快照规则

### 8.1 默认不以 version 为核心
归档层主要不是以“每天一个 version”来组织数据。

归档层默认按以下维度组织：
- 日期
- 标的
- 频度
- 分区
- 批次

也就是说，归档层更接近：
- 历史事实资产层
- 去重/补齐/merge 层
- 长期可查询资产层

而不是运行层那种“候选版本 → promote → active”的每日生产机制。

### 8.2 原始数据以去重 / 补齐 / merge 为主
归档层的主要动作是：
- merge
- upsert
- 去重
- 校正
- 补齐缺口

换言之，归档层更关注：
- 数据是否完整
- 历史是否连续
- 重跑是否幂等
- 是否可以断点恢复

而不是默认把每次运行都视为一个新的正式 version。

### 8.3 生产快照做长期保存（后置）
如果运行层形成了重要 snapshot / version，例如：
- lowfreq version
- midfreq snapshot
- report-time snapshot

则后续这些对象应沉淀到 archive，作为长期可回溯资产。

但这里再次强调：
- 这是后置项
- 不在归档层第一阶段实施
- 必须等 lowfreq 和 midfreq 本身足够稳定后再做

### 8.4 无变化不建新 version
对于归档层中**确实需要版本化**的对象，也必须坚持：
- 没变化 → 不建新 version
- 只记录 run 成功和 freshness

该规则与当前 Runtime 设计原则保持一致，目的是避免 version 噪声膨胀。

## 9. 归档层的数据库设计

归档层继续使用：
- DB：`ifa_db`
- schema：`ifa2`

**不另开新库。**

这样做的原因是：
- 便于复用当前基础设施
- 便于共享统一查询语义
- 便于后续 main 监控统一接入
- 避免在还没把架构跑稳时引入额外基础设施复杂度

建议至少包含三类表：

### 9.1 运行控制表
例如：
- `archive_jobs`
- `archive_runs`
- `archive_checkpoints`

这些表用于管理：
- 任务定义
- 每次运行
- 当前断点
- 健康状态
- 批次进度
- 恢复位置

### 9.2 数据表
归档数据按类别落表，例如：
- 历史日线
- 历史分钟
- 宏观历史
- 期货 / 商品历史
- 科技板块历史

这些表是 Archive Layer 的核心资产载体。

### 9.3 快照 / 版本沉淀表（后置）
用于保存从生产层沉淀下来的：
- lowfreq version
- midfreq snapshot
- report-time snapshot

这一类表必须存在于整体设计里，但不在第一阶段优先实施。

### 数据库设计总原则
- 归档状态必须单独可查。
- 不和 lowfreq / midfreq 的 run state 混淆。
- 尽量复用统一的 state / summary 设计范式。
- 但语义上必须明确区分 Runtime 与 Archive。

## 10. checkpoint / resume 机制

归档层必须支持**断点续跑**。

这是归档层区别于普通“一次性脚本补数”的核心能力之一。

每个归档任务至少要记录：
- `dataset_name`
- `asset_type`（股票 / 宏观 / 期货）
- `pool_name` / `scope_name`
- `backfill_start`
- `backfill_end`
- `last_completed_date`
- `shard_id` / `batch_no`
- `status`
- `records_processed`
- `error_message`
- `updated_at`

### 运行逻辑
归档层运行逻辑固定为：
- 每次只跑一个小批次
- 成功就推进 checkpoint
- 失败就保留 checkpoint
- 下次从上次停的位置继续

这意味着：
- 归档层不能设计成“一次跑完整年，失败就从头再来”
- 必须天然支持分批、断点、继续、重试
- 必须保证运行失败不会让前面的成果丢失

## 11. Runtime Daemon 和 Archiver 的关系再定义

### Runtime Daemon 负责
- lowfreq
- midfreq
- 未来 highfreq

其目标是：
- 为当前报告服务
- 跑重点对象
- 有窗口
- 有 summary
- 有 watchdog

### Archiver 负责
- 历史补齐
- 宽范围资产沉淀
- 生产快照归档（后置）
- checkpoint / resume
- 夜间运行

因此，必须明确：

**Runtime Daemon 和 Archiver 不能互相替代。**

Runtime 不应该被强行拉去做长期大回填，Archiver 也不应该被拿来承担当天报告生产任务。

系统架构上应把它们视为：
- 一个是生产运行层
- 一个是长期资产沉淀层

## 12. 归档层运行报告与监控

归档层必须有自己的运行报告。

每次运行都必须**落库**，不能只靠 stdout。

建议至少实现以下三类运行控制表：

### 12.1 archive_runs
建议字段至少包括：
- `run_id`
- `started_at`
- `completed_at`
- `window_name`
- `status`
- `total_jobs`
- `succeeded_jobs`
- `failed_jobs`
- `records_processed`
- `error_summary`

用于记录一次归档夜间窗口的总览结果。

### 12.2 archive_run_items
建议字段至少包括：
- `run_id`
- `dataset_name`
- `asset_type`
- `batch_no` / `shard_id`
- `range_start`
- `range_end`
- `status`
- `records_processed`
- `error_message`

用于记录该次运行中每个 dataset / shard / batch 的明细状态。

### 12.3 archive_checkpoints
建议字段至少包括：
- `dataset_name`
- `asset_type`
- `last_completed_date`
- `shard_id`
- `updated_at`
- `status`

用于记录“已经推进到哪里”。

### 12.4 每日文字化日报
并且，归档层必须每天固定时间自动生成一份文字化日报。

建议时间：**北京时间早上 05:00**。

日报内容至少包括：
1. 报告时间
2. 昨晚 Archiver 是否运行
3. 跑了哪些 dataset
4. 回填到哪一天
5. 成功 / 失败 / 跳过
6. 当前 checkpoint 在哪里
7. 当前 blocker 是什么
8. 是否健康

报告形式不需要 fancy：
- txt
- 纯文字
- 结构清晰

即可。

但必须基于数据库运行日志生成，而不是靠人工判断或 stdout 粘贴。

## 13. 归档层的阶段划分（最终版）

归档层按以下固定顺序推进：

### D1：归档框架
做什么：
- Archiver 基础框架
- `archive_jobs` / `archive_runs` / `archive_checkpoints`
- summary / health / watchdog
- 夜间两段窗口

目标：
- 先让 Archiver 系统跑起来

D1 的关键不是“数据面多大”，而是：
- 有任务定义
- 有运行状态
- 有断点
- 有窗口
- 有日志
- 有 health

### D2：股票历史归档
做什么：
- 股票日线归档
- 股票相关资产类历史归档
- 各板块前 10–30 活跃股的历史补齐
- 如有需要，补股票分钟历史归档

目标：
- 先把股票资产层补起来

D2 是归档层真正进入“资产沉淀”阶段的第一大块。

### D3：宏观历史归档
做什么：
- 利率
- CPI / PPI
- PMI
- GDP
- 社融 / M2
- 等重要宏观数据的历史补齐

目标：
- 做出宏观慢变量资产层

### D4：商品 / 期货 / 贵金属历史归档
做什么：
- 合约主表
- 日线
- 分钟历史（如果需要）
- 仓单
- 持仓排名
- 结算参数

目标：
- 做出商品 / 期货 / 贵金属资产层

### D5：独立运行
做什么：
- Archiver 脱离 OpenClaw 会话依赖
- 能自己持续运行
- main 可接管监控

目标：
- 进入正式运行态

D5 的标志不是“代码写完”，而是：
- Archiver 不再依赖会话上下文才能工作
- 可以作为独立系统运行
- main 能看状态、能监控、能接收日报

### D6：全量测试与验收
做什么：
- 整体验收
- checkpoint / resume 测试
- 夜间窗口测试
- 数据一致性测试
- summary / health / watchdog 测试

目标：
- 从“代码存在”变成“真正可运行系统”

### D7：生产快照资产沉淀（后置）
做什么：
- 低频 version 的长期归档
- 中频 snapshot 的长期归档
- 未来可供回放报告、做对比分析的时点资产沉淀

目标：
- 等 lowfreq 和 midfreq 成熟后，再补这一层

D7 是归档层和 Runtime 层建立更高阶衔接的阶段，但必须后置。

---

## 14. 研发顺序与执行约束

归档层后续研发必须严格按以下顺序推进：
- D1
- D2
- D3
- D4
- D5
- D6
- D7

不允许：
- 跳到 D7 提前做生产快照沉淀
- 在 D1 之前先写一堆 archive 数据逻辑
- 把 highfreq 混进归档层第一阶段
- 一边没把 Archiver 框架跑起来，一边去做大面积历史全量抓取

执行原则固定为：
1. 先把 Archiver 作为系统立住
2. 再把股票资产层补起来
3. 再扩宏观
4. 再扩商品/期货/贵金属
5. 再脱离会话依赖形成正式运行态
6. 再做全量验收
7. 最后再处理生产快照的长期沉淀

---

## 15. 与当前 iFA 总体路线的衔接

当前 iFA 总体研发顺序是：
- 低频第一阶段已基本完成
- 先做中频
- 再做高频

在这个框架下，归档层是一个**独立的长期资产层产品线**，不是中频/高频的附属脚本。

它与当前系统的关系应理解为：
- lowfreq / midfreq / future highfreq 负责生产与当日支持
- archive 负责长期资产沉淀与历史补齐

因此，后续实现中：
- 归档层必须复用当前 DB / schema / 设计范式
- 但不能与 Runtime 运行状态混淆
- 必须保留自己的 jobs / runs / checkpoints / summary / health / watchdog 语义

---

## 16. 一句话总目标

归档层的总目标是：

**把更大范围、更长时间、更完整的数据，沉淀成 iFA 自己的长期资产库；先按 D1–D7 固化与实现，先立系统、再补资产、最后再沉淀生产快照。**
